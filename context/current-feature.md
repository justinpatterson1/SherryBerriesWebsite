# Current Feature: Checkout + in-page Thank You

## Status
In Progress

## Goals
- Build a **single-page Checkout at `/checkout`** that resolves, after placing the order, into an **in-page Thank-You confirmation state** (no navigation) ‚Äî the calm last step of the same "garment" as `/cart` and `/account`, reusing the global navbar / footer / toast / theme behavior. T&T context: shipping options + payment methods are Trinidad & Tobago specific.
- **Page header:** "‚Üê Back to bag" link (‚Üí `/cart`); a **step indicator** row (‚úì Bag done ‚Üí ‚ë° Checkout active ‚Üí ‚ë¢ Confirmation upcoming, numbered circles ‚Äî active = pink-filled + blush, done = pink-tinted); 60px Italiana headline **"Secure checkout."** ("checkout" italic Playfair blush).
- **Layout** `1fr / 400px`, stacks below 940px. **Left = a form with four numbered `.co-section` cards** (pink-gradient numbered circle + Playfair heading):
  1. **Contact information** ‚Äî 2-col grid: First / Last name, then full-width Email + Phone.
  2. **Shipping address** ‚Äî full-width Address, City / Town, Landmark *(optional)*.
  3. **Shipping method** ‚Äî three custom radio cards (icon + title/sub + right-aligned price): **Pickup (Curepe)** "Collect in-store ¬∑ ready in 24h" ‚Äî **Free** (green); **TTPost** "3‚Äì5 business days nationwide" ‚Äî **$25**; **Courier Delivery** "1‚Äì2 business days ¬∑ tracked" ‚Äî **$40**. Selecting highlights (pink border + inner ring + filled dot) and updates the summary's shipping line + total.
  4. **Payment method** ‚Äî two custom radio cards: **Cash on Delivery** "Pay when your order arrives"; **Credit Card** "Secure payment via WiPay" (small WiPay tag). Choosing Credit Card smoothly reveals (max-height transition) a card-fields block ‚Äî Cardholder, Card number, Expiry (MM/YY), CVV + green-lock "Encrypted & processed securely by WiPay" note.
- **Right = sticky Order summary** (`top: 100px`, pink-bordered): scrollable line items (56px thumb w/ pink qty badge, name + variant `material ¬∑ size ¬∑ stone`, line price); a **promo row** (input + Apply ‚Üí toggles to "‚ú¶ CODE applied / ‚úï" chip); totals (Subtotal, optional **Discount** blush, **Shipping** w/ selected-method label & green "Free" for pickup, divider, big Playfair grand **Total**); full-width hot-pink **Place order** button (gold‚Üípink hover via `::before`) + green-lock "Secure checkout ¬∑ your details are protected" note.
- **Validation:** on Place order, validate First/Last/Email (regex)/Phone (‚â•7 digits)/Address/City non-empty (Landmark optional) ‚Üí red `.field-error` + per-field message, focus first bad field, toast "Please complete the highlighted fields."
- **Place order:** disable button ‚Üí "Placing your order‚Ä¶" ‚Üí on success build the order, **clear the cart + promo**, and render the **Thank-You** state.
- **Thank-You state** (replaces the checkout content, soft pink radial glow): 90px pink-gradient circle w/ white checkmark that **pops in** (scale anim); "Order confirmed" eyebrow ‚Üí big Italiana **"Thank you, {first}!"** ‚Üí "Your order is in ‚Äî we're already wrapping it in pink and gold. A confirmation has been sent to {email}."; a pink-bordered **order card** (header "Order number / SB-#####" + "Processing" status badge; rows: Date, Items, Ship to, Method, Payment, optional Discount, bold **Total**); a pink-tinted **next-steps** strip ("{eta}. We'll text {phone} with updates" ‚Äî append "‚Äî please have payment ready on arrival." for COD); CTAs **"Track my order ‚Üí"** (‚Üí `/account?view=orders`) + **"Continue shopping"** (‚Üí home `#shop` / `/products`).
- **Cart hand-off:** update `/cart`'s "Secure checkout" button (currently the mock "Redirecting‚Ä¶" toast) to **navigate to `/checkout`**.

## Notes
- **STACK TRANSLATION (critical, same as every prior feature):** the brief is written for vanilla **`checkout.html` + `checkout.js` + `styles.css`**, but this repo is **Next.js 16 / React / TS / Tailwind v4 / Prisma**. Per the Login/Cart/Wishlist/Account convention (see History), the vanilla constructs are **translated to the real stack**: a route **`src/app/checkout/page.tsx`** (server shell) + client components under **`src/components/checkout/`**, Tailwind utilities (not raw `styles.css` additions), and the **global** `Navbar`/`Footer`/providers/theme (not JS-injected chrome). The literal `checkout.html`/`checkout.js`/`styles.css` and the `data.js`/`SB.bestsellers`/`findProduct`/`ph()`/`SBToast()`/`$`-`$$` helpers will **not** be created. **WhatsApp FAB** stays deferred site-wide (consistent with cart/wishlist/account); the global footer's payment chips are not per-page restyled.
- **Cart source = real `CartProvider` + `POST /api/cart/snapshot`** (the enrichment `/cart` already uses), **not** `sessionStorage["sb-cart-rich"]`/`sb-cart`. Empty cart ‚Üí render the empty state ("Nothing to check out yet" + Shop / View bag CTAs); **no demo-seed** (dropped in cart/wishlist per precedent).
- **Promo = reuse `POST /api/promo` + the real seeded `DiscountCode`s** (`BERRY10`/`AFTERCARE15`/`WELCOME20`/`GLOWUP25`/`STUDIO50`, validated for active/expiry/usage), **not** the brief's hardcoded `SWEET10`/`BERRY20`/`FRESH15`. Persisted promo: reuse whatever `/cart` uses to carry the applied code into checkout (verify at `start`).
- **DECISIONS TO RESOLVE AT `start` (confirm before building):**
  1. **Order persistence ‚Äî the big one.** Real **`POST /api/checkout` that creates an `Order` + `OrderItem` rows** (so the order shows up in `/account` ‚Üí Orders, end-to-end with the Account feature) **vs** a client-only mock (`sessionStorage["sb-last-order"]` + `setTimeout`, matching the brief literally). *Recommend real* ‚Äî the trajectory (cart + account on real Prisma) points that way, and Account's Orders list is already wired to real `Order` data. Larger scope, so confirm.
  2. **Auth gating.** Real `Order` needs a `userId` ‚Üí checkout likely **requires sign-in** (guests ‚Üí `redirect("/login?callbackUrl=/checkout")`, mirroring `/wishlist`/`/account`). The brief doesn't mention auth; `/cart` is guest-friendly. Confirm: auth-gated checkout, or guest checkout (which forces option 1 ‚Üí mock, or a nullable-user order).
  3. **Shipping & tax model reconciliation.** This brief defines **flat T&T shipping** (pickup Free / TTPost $25 / Courier $40) and **no tax line**. `/cart` (History 2026-05-26) computes **8.75% tax + free shipping ‚â• $80 else $6**. These disagree ‚Äî decide whether checkout adopts the brief's model verbatim (and whether the cart summary should follow) or keeps cart's tax/threshold. Verify the `Order` schema's money columns (`subtotal`/`shippingCost`/`tax`/`discount`/`total`) when mapping.
  4. **Shipping address storage.** Account noted **`Order` carries no address snapshot**. Decide how the typed shipping address is persisted if option 1 is real: into `Order` columns if they exist, a related `Address` row, or a JSON snapshot ‚Äî **check `prisma/schema.prisma` at `start`** (also the `Order` `paymentMethod` / `fulfillmentStatus` / shipping-method fields + any enums to map COD / Credit Card (WiPay) / pickup-ttpost-courier onto).
  5. **Inventory + payment side-effects.** Whether placing an order **decrements product/variant inventory**, and confirm **WiPay/Credit-Card is UI-only (mock)** with no real payment integration (consistent with cart's mock checkout) ‚Äî order is created in a pending/processing state.
- **`SB-#####` order number:** map to the real `Order.orderNumber` format if creating real orders (the Account feature already renders `orderNumber`); the brief's `SB-#####` is the display convention.
- **Scope:** large multi-section feature ‚Üí at `start` outline the route/component breakdown and lock the 5 decisions above before implementing, per the "confirm before large changes" guideline.
